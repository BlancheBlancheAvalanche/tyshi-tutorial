Лекции

1. ВВЕДЕНИЕ В TYPESCRIPT

Среди инструментов, которыми каждый день оперирует фронтенд-инженер, особое положение занял TypeScript. Его появление продиктовано стремлением превратить непредсказуемую динамику JavaScript в осмысленную, предсказуемую среду. Когда кодовая база разрастается до десятков тысяч строк, следить за форматами передаваемых значений исключительно силой внимания и консольными ошибками становится рискованно. TypeScript предлагает способ заранее договориться о форме данных и получать подсказки ещё на этапе написания инструкций.

Истоки и связь с миром JavaScript

Рождение языка неразрывно сплетено с эволюцией самого JavaScript и спецификаций ECMAScript. Авторы TypeScript взяли за основу все возможности, которые стандарт ES6 и последующие версии привнесли в браузерную разработку: стрелочные функции, модули, классы, деструктуризацию. На этот каркас они наслоили необязательное описание типов, превратив свободную стилистику JS в строго регламентированную систему. Итоговый код после компиляции превращается в чистый JavaScript, способный исполняться любой средой – от устаревшего браузера до сервера Node.js. Фактически TypeScript функционирует как надмножество: любой корректный JS-файл уже является валидной программой на TS.

Судьба стандарта ECMAScript прямо отражается на возможностях TypeScript. Команда разработки внимательно следит за предложениями комитета TC39 и добавляет экспериментальные флаги для свежих синтаксических конструкций. Так, декораторы, поля классов и операторы optional chaining сначала появлялись именно в TS, а позже закреплялись в общепринятом стандарте. Это делает язык не просто статическим анализатором, а полноценным полигоном для будущих нововведений JavaScript.

Ключевые элементы языка

Система типов образует фундамент, на котором держится вся полезная нагрузка TypeScript. Примитивы, знакомые каждому JS-разработчику, здесь обретают строгие очертания: number, string, boolean, null, undefined. Для массивов используется синтаксис number[] или Array<number>. Особого упоминания заслуживает тип any – он открывает лазейку обратно в динамический мир и должен применяться осознанно, только когда точная форма данных действительно неизвестна.

Помимо базовых аннотаций язык оснащён объединениями (union) и перечислениями (enum). Первые дают разрешить переменной принимать значения нескольких заранее объявленных типов, вторые – группируют именованные константы, улучшая читаемость кода.

Интерфейсы и типовые псевдонимы служат для описания контрактов. Интерфейс задаёт перечень свойств и методов, которые обязан реализовать объект. Типовые алиасы, напротив, могут хранить не только форму объекта, но и примитивные объединения или пересечения. На практике обе конструкции часто взаимозаменяемы, однако интерфейсы удобнее расширять, а type незаменим для кортежей и сложных mapped-типов.

Дженерики выводят переиспользование на качественно иной уровень. Функция, которая умеет работать с массивом любых элементов, не теряя сведений о конкретном типе, описывается обобщённым параметром <T>. После вызова TS самостоятельно выводит, что T – это строка или число, и предохраняет от смешивания несовместимых операций.

Инструментарий и процесс сборки

Центральное звено экосистемы – компилятор tsc. Он перерабатывает файлы с расширением .ts в чистый JavaScript, попутно осуществляя проверку совместимости типов. Настройки хранятся в файле tsconfig.json; именно там включается флаг strict, активирующий самый строгий режим анализа. Дополнительно сообщество разработало серверную часть для редакторов, обеспечивающую автодополнение, мгновенную подсветку ошибок и навигацию по символам. Благодаря этому разработчик получает обратную связь ещё до запуска кода.

Сильные стороны в коллективных проектах

Когда над продуктом трудится несколько команд, контракты, зафиксированные через интерфейсы, становятся документацией, которая никогда не устаревает. Изменение сигнатуры функции сразу подсвечивает все места вызова, требуя актуализации – и это предотвращает целые классы ошибок, связанных с несовпадением ожидаемых и фактических параметров. Рефакторинг перестаёт быть лотереей: переименование свойства или метода распространяется автоматически, без боязни пропустить скрытое обращение.

Среда разработки превращается в умного ассистента. Списки доступных полей возникают после точки не наугад, а строго в соответствии с описанной моделью. Даже новичок, присоединившийся к проекту, тратит меньше времени на изучение незнакомого кода, потому что подсказки и быстрые переходы к определениям заменяют долгие поиски глазами.

Возможные сложности и границы применимости

Входной порог TypeScript действительно выше, чем у чистого JS. На начальном этапе приходится разбираться с дженериками, объединениями и сложными производными типами, что замедляет прототипирование. В маленьких скриптах или одноразовых утилитах выгода от строгой типизации не всегда окупает дополнительное время на описание контрактов. Кроме того, подключаемые сторонние библиотеки не всегда поставляются с качественными файлами деклараций, и тогда разработчик вынужден либо писать их сам, либо прибегать к any.

Несмотря на перечисленные ограничения, для средних и крупных веб-приложений TypeScript уже стал индустриальным стандартом. Он продолжает развиваться, заимствуя свежие идеи из мира функционального программирования и поддерживая самые новые версии ECMAScript, что делает его не временным увлечением, а долгосрочным спутником фронтенд-инженера.

2. ТИПЫ ДАННЫХ В TYPESCRIPT

Когда программа строится без чёткого понимания, что именно хранит каждая переменная, неизбежно появляются трудноуловимые дефекты. Система типов TypeScript призвана дать разработчику не просто справочник ограничений, а активного союзника, подсвечивающего несостыковки ещё до запуска. Правильно подобранные типы ускоряют рефакторинг, превращают редактор в интеллектуального ассистента и документируют код непосредственно в сигнатурах функций. Чтобы овладеть этим инструментом, стоит начать с каталога встроенных типов, их возможностей, граничных величин и характерных приёмов работы.

Примитивы: кирпичики любой программы

Логический тип boolean принимает строго два варианта – истину и ложь. Никакие другие значения, включая единицу или непустую строку, не будут неявно восприняты как булевы, если включён строгий режим. Основные операторы, обслуживающие этот тип, – логическое «и» (&&), «или» (||) и отрицание (!). Часто применяется конструкция двойного отрицания (!!) для явного приведения произвольного выражения к true/false.

Числовой тип number опирается на стандарт IEEE 754 с двойной точностью. Его безопасный целочисленный коридор ограничен значениями от -2⁵³+1 до 2⁵³-1, что отражают константы Number.MIN_SAFE_INTEGER и Number.MAX_SAFE_INTEGER. Выход за эти границы ведёт к потере точности, поэтому для финансовых или криптографических расчётов предусмотрен примитив bigint с суффиксом n. Арифметические операции +, -, *, /, %, возведение в степень ** и побитовые сдвиги применимы к number, однако побитовые операторы преобразуют операнды к 32-битному знаковому целому. Для округлений, тригонометрии и генерации псевдослучайных величин служит встроенный объект Math, а превращение строки в число доверяют parseInt, parseFloat и унарному плюсу.

Строка (string) представляет неизменяемую последовательность символов Unicode. Ограничение её длины теоретически упирается в 2⁵³-1 элементов, а на практике – в доступную память процесса. Строки конкатенируют плюсом, сравнивают операторами >, <, === с учётом лексикографического порядка. Богатая библиотека методов включает indexOf для поиска подстроки, slice и substring для вырезания фрагментов, startsWith/endsWith для проверки границ, replace с регулярными выражениями, toLowerCase/toUpperCase для смены регистра, trim для обрезки пробелов и split, превращающий строку в массив по разделителю.

Примитивы null и undefined обозначают отсутствие значения, но с разным смыслом: undefined обычно сигнализирует, что переменная не была инициализирована, а null – сознательную пустоту. При активном флаге strictNullChecks эти типы не подсовываются неявно туда, где ожидается, скажем, number. Такой контраст избавляет от знаменитых ошибок «Cannot read property of null». Оператор опциональной цепочки ?. позволяет безопасно обращаться к вложенным свойствам, не боясь наткнуться на пустой указатель.

Символ (symbol) гарантирует уникальность идентификатора, что полезно для скрытых свойств объектов. Он создаётся через фабрику Symbol() и не имеет литерального синтаксиса.

Структурные контейнеры и специальные формы

Массив (Array) диктует однородность элементов: заявление number[] или Array<string> обязывает все позиции содержать именно указанный тип. Динамическая длина массива ограничена лишь объёмом памяти. Арсенал методов внушителен: добавление push/pop в конец и unshift/shift в начало, трансформации map и filter, свёртка reduce, поиск find и проверка some/every. Деструктуризация и spread-оператор упрощают разбор и склейку.

Кортеж (tuple) расширяет идею массива фиксированной длиной и известными типами на конкретных индексах, например [string, number]. Так фиксируется контракт «строка, затем число»; попытка обратиться к несуществующему индексу или положить не тот тип ловится статическим анализом. Для кортежей доступны те же методы итерации, но push может нарушить предсказуемую форму, поэтому в fresh-режиме компилятор иногда ограничивает добавление разнородных элементов.

Перечисление (enum) группирует именованные константы. Числовые варианты по умолчанию стартуют с нуля, строковые требуют явной инициализации каждого члена. Enum удобен для описания конечных состояний, ролей или категорий, заменяя магические строки осмысленными метками.

Объектный тип object охватывает всё, что не относится к примитивам. Однако на практике вместо размытого object проектируют интерфейсы или типовые псевдонимы, детально описывающие форму сущности. TypeScript применяет структурную типизацию: важна не иерархия наследования, а наличие требуемых свойств.

Управление неопределённостью: any, unknown, void, never

Тип any полностью отключает проверки, возвращая разработчика в динамический мир JavaScript. Им часто злоупотребляют при миграции старого кода, однако зрелые проекты стремятся заменить его на unknown – безопасный аналог, требующий явного сужения перед использованием. С unknown нельзя вызвать метод или обратиться к полю без проверки typeof, instanceof или специального type guard.

Функция, не возвращающая значения, аннотируется void. Присвоить результат такой функции переменной можно, но значение окажется undefined. Иная ситуация с never: этот тип указывает, что управление никогда не дойдёт до конца блока (бесконечный цикл, выброс исключения). Он выступает индикатором исчерпывающей проверки в switch или условных цепочках – если анализатор видит never, значит все возможные варианты уже обработаны.

Комбинирование возможностей: объединения, пересечения и литералы

Union-тип, записываемый через вертикальную черту, позволяет величине принимать одно из нескольких очертаний: status: "loading" | "success" | "error". Сужение (narrowing) внутри условного блока выполняется операторами typeof для примитивов, instanceof для классов, оператором in для проверки наличия свойства и функциями-предикатами вида param is SomeType.

Intersection-тип, наоборот, скрещивает несколько структур в одну, требуя одновременного присутствия всех полей. Так, type Admin = User & { role: string } добавляет роль к базовой модели пользователя.

Литеральные типы фиксируют не просто категорию, а конкретное значение: направление "left", статусный код 404 или флаг true. Вместе с объединениями они моделируют размеченные союзы (discriminated unions) – мощный паттерн, где общее поле-дискриминатор определяет ветку обработки.

3. ФУНКЦИИ И СОСТАВНЫЕ ТИПЫ

Функция выступает центральным механизмом для организации повторяемой логики. В JavaScript она обладает чрезвычайной гибкостью: аргументы не проверяются, возвращаемое значение может поменять природу от вызова к вызову. TypeScript накладывает на эту пластичность жёсткий каркас, превращая сигнатуру в документированный контракт. Параллельно составные типы – массивы, кортежи – дают возможность моделировать коллекции с предсказуемой внутренней формой. Разберём, как аннотировать функции, какие возможности открывают перегрузки, чем полезны кортежи и как все эти конструкции выглядят после компиляции в чистый JS.

Аннотирование параметров и возврата

Простейшая типизация прикрепляет к каждому аргументу метку, а после скобок указывает ожидаемый результат. Рассмотрим базовый синтаксис:

  • function multiply(a: number, b: number): number { return a * b; }
  • Стрелочный вариант: const multiply = (a: number, b: number): number => a * b;

Необязательные параметры оформляются знаком вопроса. Они автоматически получают значение undefined, если вызывающая сторона их опускает:

  • function greet(name: string, salutation?: string): string { ... }

Значения по умолчанию одновременно задают тип и делают параметр факультативным – явная аннотация при этом избыточна:

  • function createId(prefix = "user"): string { ... }

Оставшиеся аргументы собираются в массив через оператор rest. TypeScript выводит его как обычный массив указанного типа:

  • function sum(...values: number[]): number { return values.reduce((a, b) => a + b, 0); }

Типизация самой функции как объекта

Функциональный тип можно вынести в отдельный псевдоним и затем применять к переменным. Это особенно ценно для колбеков:

  • type Combiner = (a: string, b: string) => string;

Сигнатура вызова (call signature) добавляет описания свойств прямо в тип функции, встречается, когда функция несёт вспомогательные поля:

  • interface DescribedFn { description: string; (x: number): boolean; }

Контекстуальная типизация работает без явных аннотаций в местах, где компилятор уже знает ожидаемую сигнатуру, например, при передаче колбека в map или addEventListener.

Перегрузки: несколько лиц одной функции

Одна и та же операция часто должна поддерживать разные наборы аргументов и выдавать согласованный результат. TypeScript позволяет объявить несколько совместимых сигнатур над общей реализацией. Реализация при этом остаётся нетипизированной относительно публичных вариантов и проверяется разработчиком вручную.

Синтаксис выглядит так:

function fetchData(url: string): Promise<string>;
function fetchData(id: number): Promise<object>;
function fetchData(param: string | number): Promise<string | object> {
  // проверка типа внутри и разная логика
}

this в аннотациях функций

Когда метод отрывается от объекта, JavaScript теряет контекст, и TypeScript позволяет предотвратить ошибку с помощью фиктивного первого параметра this:

  • function handleClick(this: HTMLElement, event: MouseEvent) { ... }

Указанная конструкция не превращается в реальный аргумент, а служит исключительно статическому анализатору и полностью исчезает после транспиляции.

Массивы: однородные последовательности

Массив в TypeScript предсказуемо хранит элементы одного вида. Синтаксис описания – квадратные скобки или обобщённая форма:

  • let names: string[];
  • let points: Array<[number, number]>;

Доступен модификатор readonly, запрещающий изменение содержимого, что полезно при передаче данных в чистые функции:

  • function display(items: readonly string[]): void { ... }

Многомерные массивы обозначаются дополнительной парой скобок, например number[][].

При компиляции в JavaScript аннотация типа полностью стирается – в исполняемом файле остаётся обычный массив без каких-либо проверок. Среда выполнения не узнает, ожидались ли там строки или числа.

Кортежи: чёткая структура каждой позиции

Кортеж расширяет массив фиксированным количеством элементов с привязанными к индексам типами. Классический пример – пара «имя, возраст»:

  • let user: [string, number];

Современный синтаксис разрешает необязательные элементы (postfix ?), остаточные элементы в конце и именованные метки, которые улучшают читаемость подсказок:

  • type Point3D = [x: number, y: number, z: number];
  • type StringNumberPair = [first: string, second: number, extra?: boolean];
  • type StringTail = [head: string, ...tail: string[]];

После обработки tsc кортеж превращается в обычный JS-массив, а все метки и ограничения длины исчезают. Дополнительный код проверки индексов не генерируется – TypeScript полагается исключительно на статический анализ.

Функции и составные типы после транспиляции

Главное правило: TypeScript удаляет все аннотации типов, интерфейсы, перечисления и дженерики. Итоговый JavaScript остаётся чистым и совместимым с выбранным целевым стандартом ES. На что обратить внимание:

  • Параметры со значениями по умолчанию и rest-аргументы сохраняются, потому что это нативный JS-синтаксис.
  • Необязательные параметры после компиляции становятся обычными; факт их опциональности проверяется только на этапе сборки.
  • Стрелочные функции при понижении версии до ES5 превращаются в обычные function-выражения, а захват this реализуется через сохранение переменной _this.
  • Перегрузки не создают множества функций – в JS остаётся единственная реализация с проверками типов внутри, которые разработчик написал сам. Компилятор не вставляет рантайм-валидацию автоматически.
  • Кортежи и readonly массивы в скомпилированном коде ничем не отличаются от простых массивов.

Такая модель позволила TypeScript заслужить доверие инженеров: сборка не раздувается вспомогательным кодом, а ошибки ловятся задолго до попадания в браузер.

4. ОБЪЕКТЫ, ИНТЕРФЕЙСЫ И ПСЕВДОНИМЫ ТИПОВ

Когда приложение насыщается сущностями с разветвлённой внутренней структурой, потребность в выразительных инструментах описания данных выходит на передний план. Примитивные аннотации вроде string или number покрывают лишь вершину айсберга; настоящая польза статического анализа раскрывается там, где TypeScript оперирует сложными формами. Язык предлагает два близких по духу, но различающихся по возможностям механизма – интерфейсы и псевдонимы типов, – а также развитую систему композиции и сужения вариантов.

Анатомия объектного описания

Простейший способ задать форму значения – инлайн-литерал типа прямо на месте объявления переменной:

  • let product: { name: string; price: number } = { name: "Laptop", price: 1200 };

Такой стиль быстро становится громоздким при повторном использовании или передаче аргументов. Чтобы избавиться от дублирования, контракт выносится в именованную конструкцию. Здесь начинается разделение на интерфейсы и псевдонимы.

Интерфейс создаёт именованный контракт и допускает расширение в будущем:

interface Merchandise {
  title: string;
  cost: number;
  description?: string;
  readonly sku: string;
}

Псевдоним типа фиксирует ту же форму через type:

type Merchandise = {
  title: string;
  cost: number;
  description?: string;
  readonly sku: string;
};

Обе формы способны содержать необязательные поля (знак вопроса), модификатор readonly для запрета перезаписи, а также индексные сигнатуры, определяющие поведение при произвольном ключе.

При поверхностном взгляде конструкции кажутся дублирующими, однако семантика и особенности слияния задают им разную роль в проектировании.

Таким образом, в типичном проекте эти инструменты сосуществуют: интерфейс описывает публичный контракт сущности, а псевдоним – вспомогательные или вычисляемые типы.

Объекты и структурная типизация

TypeScript опирается на структурную (утиную) проверку соответствия. Главенствует не происхождение объекта, а наличие ожидаемого набора полей. Это означает, что литерал, класс или фабричная функция – всё, что содержит нужные свойства, будет принято в качестве аргумента, если форма совпадает. Логика «объект подходит, если он крякает как утка» убирает необходимость в явном приведении интерфейсов и упрощает стыковку независимо написанных модулей.

Объединения типов и сужение возможных веток

Тип-объединение собирает в одну переменную несколько допустимых форм. Например, let id: string | number. При этом напрямую обратиться к методу toUpperCase компилятор запретит, ведь number такой процедуры не содержит. Здесь и вступают в дело защитники типов.

Защитники типов (type guards) представляют собой выражения, которые проверяют принадлежность значения конкретной ветке объединения. Наибольшее распространение получили:

  • Оператор typeof для примитивов: typeof value === "string".
  • Оператор instanceof для экземпляров классов: error instanceof SyntaxError.
  • Оператор in для сверки наличия свойства: "length" in data.
  • Функция-предикат, чьё возвращаемое размечено как argument is SomeType.

После прохождения такой проверки внутри блока область видимости сужается, и TypeScript разрешает обращение к членам, свойственным подтверждённой ветке.

Дискриминируемые объединения

Особую силу набирает паттерн, где у каждого варианта присутствует общее поле-литеральный признак (discriminant). TypeScript анализирует его значение и самостоятельно сужает тип:

interface TextMessage { kind: "text"; body: string }
interface ImageMessage { kind: "image"; url: string }
type AppMessage = TextMessage | ImageMessage;

function render(msg: AppMessage) {
  switch (msg.kind) {
    case "text":  return msg.body.toUpperCase();
    case "image": return `<img src="${msg.url}">`;
  }
}

Возвращаемый тип never в ветке default служит страховкой: если в объединение добавится новый вариант, а обработка не будет расширена, компилятор выдаст ошибку исчерпывающей проверки.

Пересечения: объединение свойств воедино

В отличие от объединения, пересечение требует одновременного существования всех характеристик из каждого участника. Синтаксис – амперсанд: type Admin = User & { privileges: string[] }. Результирующая сущность накапливает поля всех составляющих. Этот инструмент часто комбинируют с интерфейсами для аддитивного наращивания сигнатур.

Дженерики дают метод сочинять компоненты, не привязанные к конкретному сорту данных, но сохраняющие полное знание об их природе. Классы, реализующие интерфейсы, закрепляют архитектурные обещания. А модульная система разрезает монолит на заменяемые, тестируемые куски. Вместе они формируют фундамент, на котором возводятся надёжные и расширяемые приложения.

Классы и контракты: реализация интерфейсов

Интерфейс задаёт список полей и сигнатур, которые обязан предоставить класс. Синтаксис implements не порождает нового типа в скомпилированном коде, но принуждает TypeScript проверить полноту и согласованность реализации. Один класс способен реализовать несколько интерфейсов, собирая разрозненные обязанности под одной крышей:

interface Movable {
  speed: number;
  accelerate(delta: number): void;
}

interface Identifiable {
  readonly id: string;
}

class Vehicle implements Movable, Identifiable {
  constructor(
    public speed: number,
    public id: string
  ) {}
  accelerate(delta: number) { this.speed += delta; }
}

Ключевые моменты:

  • Поля могут быть описаны в интерфейсе как readonly или с модификатором ?; класс обязан их предъявить.
  • Методы должны совпадать по сигнатуре, включая параметры и возвращаемый тип.
  • Модификаторы доступа (private, protected, public) в интерфейсе не указываются; это забота реализации.
  • При компиляции конструкция implements полностью исчезает – JavaScript не хранит сведений о соответствии контракту.

Абстрактные классы занимают нишу между готовой реализацией и чистым интерфейсом. Они могут содержать воплощённые методы и поля, а также помечать часть поведения словом abstract, требуя от наследников довести детали до ума. В отличие от интерфейса, абстрактный класс способен иметь конструктор и модификаторы доступа, генерируя реальный JS-код.

Модули: нарезка кода на файлы

TypeScript перенял синтаксис ES-модулей: export и import. Любой файл, содержащий хотя бы одну такую инструкцию, воспринимается компилятором как модуль с собственной областью видимости. Имена внутри не загрязняют глобальное пространство, пока их явно не экспортируют. Варианты экспорта:

  • Именованный: export function parse(data: string) { ... } либо export { parse, render }.
  • По умолчанию: export default class App { ... }.

Настройка module в tsconfig.json определяет, во что превратятся директивы импорта-экспорта после сборки: CommonJS, AMD, UMD, ES2015/ESNext. Это даёт контроль над целевой средой выполнения. Современные проекты всё чаще ориентируются на ESNext, передавая задачу финальной упаковки бандлерам вроде Webpack или Vite.

Помимо ES-модулей TypeScript понимает устаревшие неймспейсы (namespace), группирующие логику внутри одного файла. Их применение сегодня ограничено, так как модули решают ту же задачу более гибко и совместимо с экосистемой.

Проблема бездженериковой универсальности

Когда пишут функцию, принимающую любое значение и возвращающую точно такое же, без дженериков выбор сводится либо к дублированию под каждый тип, либо к замыливанию сигнатуры с помощью any. Второй путь отключает проверку: TypeScript забывает, что входило в аргумент, и разрешает вызывать на результате несуществующие методы. Страдает автодополнение, а ошибки всплывают лишь во время исполнения. Дженерики восполняют этот пробел, привязывая тип выхода к типу входа без потери информации.

Дженерики в действии: функции, интерфейсы, классы

Обобщённая функция захватывает тип через угловые скобки, используя параметр <T> как заполнитель. При вызове вывод срабатывает автоматически, хотя разработчик может зафиксировать тип вручную:

function identity<T>(value: T): T {
  return value;
}
const numberIdentity = identity(42);        // тип 42
const explicit = identity<string>("text"); // T = string

Интерфейсы и типовые псевдонимы также обретают ти́повые параметры, позволяя моделировать контейнеры:

interface Repository<T> {
  items: T[];
  findById(id: string): T | undefined;
}

Классы с дженериками – популярный приём для коллекций, сервисов и фабрик:

class Stack<T> {
  private data: T[] = [];
  push(item: T) { this.data.push(item); }
  pop(): T | undefined { return this.data.pop(); }
}

Ограничения (constraints) сужают множество допустимых форм для T. Синтаксис extends заставляет параметр удовлетворять определённому контракту. Классический пример – доступ к свойству length:

interface Lengthwise {
  length: number;
}

function logLength<T extends Lengthwise>(arg: T): number {
  return arg.length;
}
logLength("hello");   // OK
logLength([1,2,3]);   // OK
logLength(42);        // Ошибка: число не имеет length

Продвинутые приёмы ООП

TypeScript насыщает классическое прототипное наследование строгими модификаторами доступа: public (по умолчанию), protected (видимость внутри класса и потомков), private (только внутри объявившего класса). Синтаксис параметров-свойств в конструкторе позволяет объявить и сразу инициализировать члены, сокращая шаблонный код:

class Point {
  constructor(
    public x: number,
    protected y: number,
    private readonly tag: string
  ) {}
}

Наследование (extends) позволяет строить иерархии сущностей, а ключевое слово super – вызывать родительские конструкторы и методы. Абстрактные классы и защищённые конструкторы дают тонкий контроль над созданием экземпляров, запрещая прямое инстанцирование.

Итоговый штрих: после прохода компилятора все типовые параметры, интерфейсы и модификаторы доступа стираются. В полученном JavaScript остаются лишь чистые функции, классы с прототипным наследованием и модульные конструкции, соответствующие выбранной цели сборки. Такой подход сохраняет производительность рантайма и привычную среду исполнения, перемещая всю тяжесть верификации на этап написания кода.

Укрощение DOM: элементы и их свойства

Браузер хранит страницу в виде дерева объектов, и корень этого дерева – HTMLElement. От него расходятся специализированные ветви: HTMLInputElement, HTMLButtonElement, HTMLDivElement. Простое обращение document.querySelector('.field') возвращает объединение Element | null, потому что компилятор не угадывает, какой именно тег спрятан за селектором. Чтобы открыть доступ к специфическим свойствам – value у инпута, checked у чекбокса – применяют приведение типа через as или уточняющие проверки.

Асинхронность под надзором

Сетевые запросы, таймеры, чтение файлов – операции, разворачивающиеся вне основного потока. TypeScript оборачивает их в дженерик Promise<T>, привязывая разрешаемое значение к конкретной форме. Функция, помеченная async, неявно заворачивает результат в промис, и компилятор выводит этот промис автоматически:

async function fetchProfile(id: number): Promise<UserProfile> {
  const response = await fetch(`/api/user/${id}`);
  return response.json(); // выведено как Promise<UserProfile>
}

Обработка ошибок в async-контексте требует аккуратности: блок catch получает аргумент типа unknown, а не Error, потому что выбросить можно и строку, и число. Приведение через instanceof Error становится обязательной практикой, и TypeScript подсказывает, что поле message доступно только после проверки.

Общение с сервером: типизация запросов и ответов

Самый частый канал обмена – HTTP, и наиболее уязвимое место – обработка JSON, которую сервер выдаёт без каких-либо гарантий соответствия схеме. Типичный сценарий начинается с объявления интерфейса, описывающего ожидаемую структуру:

interface Order {
  id: string;
  amount: number;
  status: 'pending' | 'shipped';
}

Функция-обёртка над fetch подменяет тип возвращаемого значения, указывая конкретную модель. Однако компилятор верит объявлению безоговорочно: если сервер в ответе пропустил поле или изменил формат, TypeScript этого не заметит. Поэтому набирает популярность подход с двойным барьером: сначала результат помечается как unknown, затем библиотека-валидатор (Zod, io-ts) проверяет сырые данные и сужает тип до целевого. Так описание контракта начинает работать и на этапе компиляции, и в рантайме:

const raw: unknown = await response.json();
const parsed = OrderSchema.parse(raw); // теперь parsed: Order

Для API с множеством эндпоинтов объявляют централизованный файл с интерфейсами запросов и ответов. Методы вроде GET, POST, PUT получают свои параметры и тела, описанные через отдельные псевдонимы. Такой реестр становится живой документацией, доступной любому члену команды.

Подключение внешних библиотек

Современная экосистема npm поставляет огромный массив готовых решений. Часть пакетов уже написана на TypeScript и распространяется с файлами .d.ts, генерируемыми при публикации, – установка такого пакета не требует дополнительных шагов. Другие же созданы на чистом JavaScript, и без дополнительных аннотаций импорт будет проходить под типом any или вовсе вызывать ошибку. На помощь приходит репозиторий DefinitelyTyped и команда npm i --save-dev @types/package-name. Эти пакеты содержат исключительно файлы деклараций, связывающие вызовы библиотеки с её типами.

Если готовых типов не обнаружилось ни в самом пакете, ни в DefinitelyTyped, разработчик может создать локальный ambient-файл (*.d.ts), в котором опишет используемые функции и объекты с помощью ключевого слова declare. Минимальная декларация для утилиты, экспортирующей одну функцию, выглядит так:

declare module 'untyped-lib' {
  export function compute(a: number, b: number): number;
}

Благодаря этому TypeScript признаёт модуль, и автодополнение начинает работать. Постепенное наполнение декларациями позволяет типизировать проект шаг за шагом, не дожидаясь полного покрытия. Для популярных библиотек сообщество активно поддерживает типовые пакеты, поэтому спектр совместимых решений постоянно расширяется.

Итог: TypeScript преобразует хаотичное браузерное окружение в территорию с чёткими правилами. DOM-дерево перестаёт быть набором безымянных узлов, асинхронные потоки обретают предсказуемые очертания, а внешний код – внятные сигнатуры. Вся эта разметка исчезает при сборке, но оставляет за собой привычный, быстрый JavaScript и спокойствие инженера, который больше не гадает, что именно прячется за переменной.

load(): void {
  const data = localStorage.getItem("tasks");

      break;

      break;

      break;

      result = prev / current;
      break;
    default:

  this.operator = null;
  this.previousInput = '';
  this.updateDisplay();
}

Шаг 7. Реализация класса Point
Змейка будет состоять из сегментов. Каждый сегмент — это объект класса Point, который знает свои координаты и имеет ссылку на следующий за ним сегмент (хвост).
Добавьте в snake.ts определение класса Point:

class Point {
  x: number;
  y: number;
  width: number = 10;
  height: number = 10;
  tail: Point | null;

  constructor(x: number, y: number) {
    this.x = x;
    this.y = y;
    this.tail = null;
  }

  moveToPoint(p: Point): void {
    if (this.tail != null) {
      this.tail.moveToPoint(this);
    }
    this.x = p.x;
    this.y = p.y;
  }
}
Обратите внимание на ключевое свойство tail: Point | null. Это рекурсивная структура данных: голова указывает на первый сегмент, тот — на следующий, и так до null. Метод moveToPoint заставляет каждый сегмент переместиться на место предыдущего, создавая иллюзию движения. Это ключевая логика!

Шаг 8. Создание скелета класса SnakeGame

class SnakeGame {
  canvas: HTMLCanvasElement;
  context: CanvasRenderingContext2D;
  snakeHead: Point | null;
  food: Point | null;
  direction: Direction;
  tileSize: number;

  constructor(canvas: HTMLCanvasElement) {
    this.canvas = canvas;
    this.canvas.width = 400;
    this.canvas.height = 400;
    this.context = this.canvas.getContext("2d")!;
    this.snakeHead = null;
    this.food = null;
    this.direction = Direction.Right;
    this.tileSize = 10;

В конструкторе мы сохраняем переданный элемент canvas, задаём ему размеры и получаем context для рисования. Восклицательный знак ! (non-null assertion) говорит TypeScript, что мы уверены, что getContext("2d") не вернёт null (это стандартная практика при работе с canvas).

Шаг 11. Реализация отрисовки игры
Теперь самое интересное — визуализация. Метод draw должен пробегаться по всей змейке и рисовать каждый её сегмент, а также еду.
Замените заглушки и добавьте метод draw в класс SnakeGame:

draw(): void {
  this.context.clearRect(0, 0, this.canvas.width, this.canvas.height);
  this.context.fillStyle = "#000000";

  var p = this.snakeHead;
  while (p != null) {
    this.context.fillRect(p.x, p.y, p.width - 1, p.height - 1);
    p = p.tail;

  if (this.food) {
    this.context.fillStyle = "#ff0000";
    this.context.fillRect(this.food.x, this.food.y, this.food.width - 1, this.food.height - 1);
  }
  this.context.fillStyle = "#000000";
}
Чтобы этот метод вызывался, добавьте его вызов в игровой цикл — метод loop:

Шаг 12. Реализация движения змейки
Змейка должна двигаться. Реализуем логику перемещения головы в текущем направлении.
Добавьте метод move в класс SnakeGame:

move(): void {
  if (!this.snakeHead || !this.food) return;
  var xNew = this.snakeHead.x;
  var yNew = this.snakeHead.y;
  switch (this.direction) {
    case Direction.Left: xNew -= this.tileSize; break;
    case Direction.Right: xNew += this.tileSize; break;
    case Direction.Up: yNew -= this.tileSize; break;
    case Direction.Down: yNew += this.tileSize; break;
  }
  // Движение хвоста
  if (this.snakeHead.tail) {
    this.snakeHead.tail.moveToPoint(this.snakeHead);
  }
  this.snakeHead.x = xNew;
  this.snakeHead.y = yNew;
}
И вызовите этот метод в loop() перед this.draw();. Порядок важен: сначала меняем состояние (двигаем), потом рисуем.
loop(): void {
  this.move();
  this.draw();
}
Скомпилируйте и проверьте в браузере. Змейка должна поползти вправо и исчезнуть за границей холста. Игра пока не проверяет столкновения.

Шаг 13. Обработка столкновений: границы
Змейка не должна уползать за пределы поля. Напишем проверку на столкновение с границами.
Добавьте в класс SnakeGame метод snakeOutsideBounds:

snakeOutsideBounds(): boolean {
  if (!this.snakeHead) return true;
  return (this.snakeHead.x < 0 ||
    this.snakeHead.y < 0 ||
    this.snakeHead.x + this.snakeHead.width > this.canvas.width ||
    this.snakeHead.y + this.snakeHead.height > this.canvas.height);
}
Этот метод проверяет, выходят ли координаты головы за пределы холста 400x400.

Шаг 14. Обработка столкновений: сама с собой
Вторая причина окончания игры — змейка врезается в свой хвост.
Добавьте в класс SnakeGame метод snakeTouchingItself:

snakeTouchingItself(): boolean {
  if (!this.snakeHead) return false;
  var p = this.snakeHead.tail;
  while (p != null) {
    if (p.x == this.snakeHead.x && p.y == this.snakeHead.y) return true;
    p = p.tail;
  }
  return false;
}
Логика проста: мы пробегаем по всем сегментам хвоста и сравниваем их координаты с координатами головы.

Шаг 17. Размещение еды (метод placeFood)
Реализуем логику случайного размещения еды. Еда не должна появляться на теле змейки.
Найдите метод-заглушку placeFood() и замените её полной реализацией:

placeFood(): void {
  if (!this.snakeHead) return;
  const tilesX = this.canvas.width / this.tileSize;
  const tilesY = this.canvas.height / this.tileSize;
  const noOfTiles = tilesX * tilesY;
  var a: Point[] = new Array(noOfTiles);

  // Создаем массив ВСЕХ возможных клеток
  for (var i = 0; i < noOfTiles; i++) {
    var x = (i % tilesX) * this.tileSize;
    var y = (Math.floor(i / tilesX)) * this.tileSize;
    a[i] = new Point(x, y);
  }
  // Получаем массив клеток, занятых змейкой
  var snakeParts: Point[] = new Array();
  var p = this.snakeHead;
  while (p != null) {
    snakeParts.push(p);
    p = p.tail!;
  }
  // Оставляем только свободные клетки
  var validPoints: Point[] = new Array();
  for (var i = 0; i < a.length; i++) {
    if (!this.pointInArray(a[i], snakeParts)) validPoints.push(a[i]);
  }
  // Выбираем случайную свободную клетку
  if (validPoints.length > 0) {
    var newPointIndex = randomInt(validPoints.length - 1, 0);
    this.food = validPoints[newPointIndex];
  }
}

pointInArray(p: Point, pointArray: Point[]): boolean {
  for (var i = 0; i < pointArray.length; i++) {
    var pointAtIndex = pointArray[i];
    if (pointAtIndex.x == p.x && pointAtIndex.y == p.y) return true;
  }
  return false;
}
Теперь при вызове restart() будет появляться красный квадратик еды. Но змейка пока не умеет её есть.

Шаг 18. Поедание еды и рост змейки
Самая приятная механика: змейка должна расти, когда её голова попадает на клетку с едой.
Модифицируйте метод move(). Найдите условие, где вычисляются xNew и yNew. Добавьте проверку на совпадение с координатами еды:

move(): void {

Теперь реализуйте метод eat() в классе SnakeGame:
eat(): void {
  if (!this.food || !this.snakeHead) return;

  newHead.tail = this.snakeHead;
  this.snakeHead = newHead;

Шаг 21. Обработка ошибок с union-типами
Наш код полон проверок if (!this.snakeHead) return;. Сделаем одну из них более явной с помощью гварда типов.
В методе eat() мы можем объединить проверку в более читаемую:

private static rotateClockwise(arr: number[][]): number[][] {
  let transformedArray: number[][] = [];
  const M = arr.length;
  const N = arr[0].length;

  for (let y = 0; y < N; y++) {
    transformedArray.push([]);
    for (let x = 0; x < M; x++) {

    }
  }

  for (let y = 0; y < M; y++) {

    }
  }
  return transformedArray;
}
